# 最適化結果の急変原因診断プラン

## 問題の整理

### 観察された事実
1. **Train/Test乖離**: Train 1.71% → Test -3.43%（約5.14%ポイントの乖離）
2. **1/21→1/29の急変**: 同じStudy Aで、1/21は4.29%（勝率91.67%）→ 1/29は-3.43%（勝率25%）
3. **パラメータの違い**: 1/21はBB逆張り、1/29はBB順張り

### 疑うべき原因（優先順位）

#### A. 評価窓（eval_end）の実装がズレている（最優先）
**もし全ポートフォリオが2024-12-31まで評価されているなら、固定ホライズンが崩れている**

#### B. 1/21の"良いbest_params"が探索空間から外れた
**探索空間/実装/データ/乱数/並列の影響で、良い領域に到達できていない**

#### C. データのスナップショットが変わった
**株価・財務データの後日修正や欠損埋めで結果が変わる**

#### D. 目的関数（mean）由来の不安定さ
**外れ値に弱いが、今回の主因かは微妙**

---

## 診断手順（3本勝負）

### ① 固定ホライズンが本当に守れているか（ログで1発）

**確認方法**:
1. 最適化実行時のログで「チェックA」を確認
2. または、`scripts/reevaluate_best_params.py`を実行してログを確認

**期待される挙動**:
- 2022-01-31 の eval_end ≈ 2024-01-31
- 2022-12-30 の eval_end ≈ 2024-12-30

**もし全部 2024-12-31 なら**:
→ 固定ホライズンが崩れていて、結果比較が無意味（最優先で修正が必要）

**確認コマンド**:
```bash
python scripts/reevaluate_best_params.py
```

このスクリプトは以下を出力します：
- 各rebalance_dateのeval_end_date
- holding_years
- チェックA/Bの詳細ログ

---

### ② 1/21のbest_paramsを"今のコード/データ"で再評価（最も情報量が大きい）

**目的**: ケースA/Bの判定

**ケースA：1/21 bestが今でも良い**
→ 戦略の"良い領域"は存在するが、**1/29の最適化がそこに到達できていない**
→ 原因は「探索空間の変更/サンプラーの挙動/並列/乱数/制約」で確定方向

**ケースB：1/21 bestが今では悪い**
→ 原因は「データ更新」「評価窓の実装差」「スコア/選定ロジックが変わった」など
→ **最適化以前の問題**が濃厚

**実行コマンド**:
```bash
python scripts/reevaluate_best_params.py
```

このスクリプトは以下を出力します：
- Train期間での再評価結果
- Test期間での再評価結果
- 元の結果（1/21）との比較
- ケースA/Bの判定

---

### ③ "同じ設定"で1/21の再現実行（seed/並列を固定）

**目的**: 再現性の確認

**実行方法**:
```bash
python -m omanta_3rd.jobs.optimize_longterm \
  --start 2018-01-31 \
  --end 2024-12-31 \
  --study-type A \
  --n-trials 200 \
  --train-end-date 2023-12-31 \
  --as-of-date 2024-12-31 \
  --horizon-months 24 \
  --lambda-penalty 0.00 \
  --objective-type mean \
  --n-jobs 1 \
  --bt-workers 1 \
  --random-seed 42
```

**注意点**:
- `n_jobs=1`, `bt_workers=1`で並列化を無効化（再現性優先）
- `random_seed=42`で乱数を固定
- データキャッシュを同じ条件で扱う（再構築の要否を統一）

**再現できない場合**:
→ 並列や外部状態（データ/キャッシュ）が結果を支配している可能性が高い

## 追加の診断ログ（次にハマりにくくする）

以下の情報をログに出力すると、原因追跡が楽になります：

### 実装済み
- ✅ チェックA: 固定ホライズン評価の確認
- ✅ チェックB: 未来リーク確認
- ✅ 各rebalance_dateのeval_end_dateとholding_years

### 追加推奨（将来の改善）
- `logic_version`（コミットIDなど）
- `data_snapshot_id`（データ更新日や件数ハッシュ）
- 各rebalanceでの:
  - フィルタ後候補数（流動性/ROE/…の各段階）
  - `target_min`を割ってセクター制限緩和が発動した回数
  - `selected_codes`と`weights`のhash

## 次のステップ

1. **まず①を実行**: 固定ホライズン評価が正しく機能しているか確認
2. **次に②を実行**: 1/21のbest_paramsを再評価してケースA/Bを判定
3. **判定結果に応じて**:
   - **ケースAの場合**: 探索空間/サンプラー/並列/乱数の問題を調査
   - **ケースBの場合**: データ更新/評価窓の実装差/スコア/選定ロジックの変更を調査
4. **必要に応じて③を実行**: 再現性の確認

---

## 参考: 探索範囲を絞る場合の判断

**②で「1/21 bestが今でも良い」なら**:
→ 絞る価値が高い（局所探索フェーズへ）
- 1/21 bestの近傍レンジで再最適化（±幅を明示）
- Optunaの`enqueue_trial`で1/21 bestを初期点として投入

**②で「1/21 bestが今は悪い」なら**:
→ 範囲を絞る前に、評価設計/データ/ロジック差分を直すべき
（絞っても同じ悪さを再生産するだけ）
