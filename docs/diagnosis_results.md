## 原因の特定

### 確定したこと
1. **評価設計は正しい**: 固定ホライズン評価と未来リーク防止は正常に機能している
2. **データ更新の影響は小さい**: 1/21のbest_paramsが現在でも良い結果を出している
3. **戦略の「良い領域」は存在する**: 1/21のbest_paramsは現在でも有効

### 疑われる原因（優先順位）
1. **探索空間の変更**: 1/29の最適化で、1/21のbest_paramsの近傍を探索できていない
2. **サンプラーの挙動**: TPEサンプラーが1/21のbest_paramsの近傍を探索していない
3. **並列化の影響**: 並列実行による探索順序の変化
4. **乱数の影響**: シード固定でも並列化により探索順序が変わる可能性
5. **制約の変更**: RSI/BB方向のcategorical化により探索空間が変わった

## 次のステップ

### 1. 1/21のbest_paramsを初期点として投入（最優先）

Optunaの`enqueue_trial`を使用して、1/21のbest_paramsを初期点として投入する。

**メリット**:
- 「良い谷」に確実に到達できる
- 探索の初期段階から良い領域を探索できる
- 局所探索フェーズに移行できる

**実装方法**:
```python
# 1/21のbest_paramsを読み込む
best_params_1_21 = load_best_params("optimization_result_optimization_longterm_studyA_20260121_204615.json")

# Optunaのstudyにenqueue
study.enqueue_trial(best_params_1_21["best_params"])
```

### 2. 探索空間を1/21のbest_paramsの近傍に絞る（局所探索）

1/21のbest_paramsの各パラメータに対して、±幅を設定して探索範囲を絞る。

**例**:
- `w_quality`: 0.44 ± 0.10 → [0.34, 0.54]
- `rsi_base`: 78.99 ± 10.0 → [68.99, 88.99]
- など

**注意**: 範囲を絞りすぎると、最適解を見逃す可能性があるため、適度な幅を設定する。

### 3. 並列化設定の見直し（再現性向上）

- `n_jobs=1`, `bt_workers=1`で再現性を確認
- 必要に応じて、並列数を調整

### 4. 目的関数の改善（median/trimmed_mean）

外れ値に強い目的関数を使用することで、より安定した最適化が可能。

**実装済み**: `objective_type`パラメータで`median`/`trimmed_mean`を選択可能
